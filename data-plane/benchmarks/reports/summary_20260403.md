## Key Findings

### Fastest paths
- **Narrative collection** is near-instant (180ms median). T1 accumulates context silently; only T2 fires an LLM call.
- **Single scheduling confirmation** (`scheduling_slot_confirm_speak`) is the fastest individual LLM call at **163ms avg / 154ms median**.
- **FAQ deflect** (legal question, no `faq_answer` call) is the fastest FAQ turn at **201ms avg**.

### Slowest / highest-variance paths
- **`dc_extract`** is the most variable call: **493ms avg, 2535ms max** (5× the median). This is the dc_extract LLM call for extracting name/email — warm-up or cold-cache spikes are visible in runs 1 and 4.
- **`dc_confirm_classify`** shows similar spike behaviour: **314ms avg, 1703ms max**. Together these make `data_collection` the highest-variance scenario.
- **Planner** adds ~378ms per turn, which compounds to **2272ms for a 6-turn call**. This overhead applies to every agent invocation in the real pipeline.

## Observations & Opportunities

1. **`dc_extract` spikes** (up to 2535ms) suggest occasional model cold-start or token-budget pressure on the extraction prompt. Worth monitoring — if consistent in production, consider trimming the dc_extract prompt or pre-warming.

2. **Planner tax is ~378ms per turn.** For a full call (data_collection → narrative → scheduling ≈ 13 turns), the planner alone adds ~4.9 seconds of total latency. If we can skip the planner for unambiguous single-stage turns, this is the highest-ROI optimisation.

3. **FAQ is very consistent** (1260–1662ms range across 10 runs, IQR tight). Multi-question turns (`faq_multi_match` + `faq_answer`) average ~440ms; legal-deflect turns are 201ms.

4. **Scheduling is the most efficient multi-turn agent** at 844ms avg for a full 3-turn booking flow. The slot-choice turn (2 LLM calls) is the only compound call and stays under 400ms median.

5. **Overhead ≈ 0ms across all scenarios.** Python routing, state management, and JSON parsing add negligible time — all latency is pure LLM.

## Planner Warning

2 of 10 planner runs logged:
```
Intent mapping produced no invoke steps — using fallback
```
This is the planner falling back when it can't map an utterance to a workflow step. No visible latency impact in this benchmark, but worth tracking in production — a fallback plan on an ambiguous mid-call utterance will delay agent selection.
